# session-log skill

## 1. 目标

创建一个 `session-log` skill，用于在**当前工作目录**下维护一个 `session-log.md` 文件，持续记录用户与助手之间的任务协作过程，方便回顾问题、解决方案和当前整体进度。

### 1.1适用人群

想要进行小型项目开发、记录开发流程、写博客、沉淀自己知识库的人群，输出的session-log文档可以配合notion使用，搭建自己的知识库。

---

## 2. 核心功能

### 2.1 文件创建与定位

- 日志文件固定命名为：`session-log.md`
- 存放位置：**当前工作目录**
- 若文件不存在，自动创建
- 若文件已存在，则继续在原文件中**追加**内容，不覆盖历史记录

### 2.2 持续追加记录

- 每次助手完成一轮**有效任务协作回复**后，可向 `session-log.md` 追加一条新的日志记录
- 记录方式为**结构化摘要**
- 不要求逐字转录完整对话，只记录关键内容

### 2.3 文件头部概览

在文件头部维护一个“**会话概览**”区域，用于展示当前整体进度，而不是某一轮对话全文。

概览至少包含：

- 创建时间
- 最近更新时间
- 当前项目目录
- 当前状态（进行中 / 已完成 / 阻塞中 / 部分完成）
- 当前阶段
- 累计记录数

---

## 3. 触发方式区分

本 skill 同时面向 **Codex** 和 **Claude Code** 两种环境，但两者的触发方式不同，必须明确区分。

### 3.1 Codex 触发方式

Codex 环境中，`/xxx` 会被客户端内置命令系统优先解析，因此**不能依赖自定义 `/session-log` 命令**触发本 skill。

Codex 中应采用：

- **显式 skill 调用方式**：`$session-log ...`
- 或自然语言方式：`使用 session-log skill ...`

推荐写法：

- `$session-log 初始化日志`
- `$session-log 更新概览`
- `$session-log 查看状态`
- `$session-log 关闭记录`

在 Codex 中，后续动作说明本质上属于**提示词内容**，而不是系统级原生命令。

### 3.2 Claude Code 触发方式

Claude Code 支持自定义 slash command，因此可使用真正的 `/` 命令触发。

推荐写法：

- `/session-log init`
- `/session-log update-ov`
- `/session-log status`
- `/session-log off`

其中：

- 命令名为：`/session-log`
- 动作为其后参数：`init`、`update-ov`、`status`、`off`

### 3.3 统一设计原则

虽然 Codex 与 Claude Code 的触发语法不同，但两者的**业务语义应保持一致**：

- 初始化日志
- 更新概览
- 查看状态
- 关闭记录

也就是说：

- Codex 偏向：`$session-log + 动作描述`
- Claude Code 偏向：`/session-log + 动作参数`

## 4. 动作定义

### 4.1 初始化日志

语义：

- 创建或打开当前目录下的 `session-log.md`
- 若文件不存在，则初始化文件结构
- 开启当前会话的记录模式

Codex 推荐触发：

- `$session-log 初始化日志`

Claude Code 推荐触发：

- `/session-log init`

### 4.2 更新概览

语义：

- 只更新文件头部概览
- 不要求单独追加正文
- 概览内容反映截至当前的整体进度，而非最后一轮对话全文

Codex 推荐触发：

- `$session-log 更新概览`

Claude Code 推荐触发：

- `/session-log update-ov`

### 4.3 查看状态

语义：

- 查看当前日志文件路径
- 查看文件是否存在
- 查看概览是否存在
- 查看累计记录数、最近更新时间、当前状态、当前阶段
- 结合当前会话上下文判断记录模式是否已开启

Codex 推荐触发：

- `$session-log 查看状态`

Claude Code 推荐触发：

- `/session-log status`

### 4.4 关闭记录

语义：

- 关闭当前会话的自动记录模式
- 后续对话不再自动写入正文
- 不自动修改日志文件，除非用户另行要求

Codex 推荐触发：

- `$session-log 关闭记录`

Claude Code 推荐触发：

- `/session-log off`

---

## 5. 每条日志记录的内容

每次追加的日志条目应包含以下字段：

1. **任务目标**  
   当前这一轮要解决什么问题

2. **我让你做的事情**  
   助手要求用户执行的动作、步骤或确认事项

3. **你遇到的问题**  
   用户在执行中遇到的报错、阻塞、理解偏差或环境问题

4. **我给出的解决方式**  
   助手提供的排查思路、修复步骤、建议方案

5. **你的反馈**  
   用户对建议的反馈、确认、补充信息或执行结果

6. **实际最终解决方式**  
   最终真正采用的方案或处理结果

7. **当前状态**  
   - 已解决
   - 部分解决
   - 未解决

8. **遗留问题 / 下一步**  
   若本轮未完全结束，记录后续待办或下一步建议

9. **关键命令 / 报错 / 文件**  
   如有实际操作，记录重要命令、报错摘要、涉及文件路径或修改结果

## 6. 记录规则

### 6.1 追加原则

- 只追加，不覆盖历史正文
- 如需更新整体状态，只更新头部概览中的汇总字段

### 6.2 摘要原则

- 以简洁、可回顾、可检索为目标
- 避免重复啰嗦
- 重点保留任务推进信息，而非闲聊内容

### 6.3 更新时间

- 每次追加新记录时：
  - 更新“最近更新时间”
  - 更新“累计记录数”
  - 根据最新进展更新“当前状态”和“当前阶段”

### 6.4 概览更新原则

- 会话概览**不要求每次普通对话都更新**
- 概览仅在用户明确要求更新概览时更新
- 概览内容应反映**截至当前的整体工作进度**
- 概览不是最后一轮对话全文，而是整体状态摘要

### 6.5 自动记录范围

- 开启记录模式后，自动记录的对象应限于**任务相关、能够推动问题解决的有效对话**
- 纯闲聊、简单确认、无实质推进的内容可不记录
- 关闭记录模式后停止自动追加正文记录

### 6.6 不再采用的触发设计

- 在 Codex 中，不再将 `/session-log`、`/session-log:update-ov`、`/session-log:status`、`/session-log:off` 视为可直接输入的真实命令
- 这些写法仅保留其**动作语义参考价值**
- 在 Claude Code 中，采用 `/session-log <action>` 形式替代旧式冒号命名

## 7. Markdown 格式要求

### 7.1 文件结构

文件采用如下结构：

```md
# Session Log

<!-- SESSION_OVERVIEW_START -->
## 会话概览
- 创建时间：
- 最近更新时间：
- 当前项目目录：
- 当前状态：
- 当前阶段：
- 累计记录数：
<!-- SESSION_OVERVIEW_END -->

---

<!-- SESSION_ENTRIES_START -->
## YYYY-MM-DD HH:mm:ss

- 任务目标：
- 我让你做的事情：
- 你遇到的问题：
- 我给出的解决方式：
- 你的反馈：
- 实际最终解决方式：
- 当前状态：
- 遗留问题 / 下一步：
- 关键命令 / 报错 / 文件：
<!-- SESSION_ENTRIES_END -->
```

### 7.2 标题层级

- 文件标题固定为 `# Session Log`
- 每轮日志使用二级标题 `## 时间戳`

### 7.3 语言与编码要求

- 日志文档的主要阅读者惯用语言为**中文**
- 日志内容、字段说明、状态描述、补充说明应默认使用**简体中文**
- 写入文件时应避免出现乱码、异常字符、错误编码或中英文混乱显示
- 文件应采用可稳定显示中文的编码方式保存，确保在常见编辑器中打开时可正常阅读
- 如涉及命令、路径、报错原文等必须保留英文内容，可按原文保留，但说明文字仍应使用中文

---

## 8. 安全与隐私要求

- 不主动记录密码、token、密钥、Cookie、隐私数据等敏感信息
- 若对话中出现敏感信息，仅记录**脱敏后的摘要**
- 对报错和命令也尽量保留必要上下文，避免泄露敏感值
- 在处理中文内容时，应优先保证文本可读性和编码稳定性，避免因写入方式导致 Markdown 文档出现乱码

---

## 9. 适用场景

该 skill 适用于：

- 开发排错记录
- 项目协作追踪
- 会话过程留档
- 阶段性任务总结
- 长流程问题处理复盘

---

## 10. 推荐行为补充

建议 skill 在写入日志时遵循以下策略：

- 仅在**任务相关对话**中记录，不必记录纯闲聊
- 如果一轮对话没有明确推进，可选择不写入
- 如本轮内容很少，可简要记录，不强行填满所有字段
- 若某字段无内容，可写“无”或省略为空值
- 开启记录模式后，正文可自动持续记录；概览仍保持手动更新
- 不设置单独的 `record` 子动作，避免与自动记录模式冲突

## 11. 成功标准

如果 skill 正常工作，应满足：

- 在当前目录自动创建 `session-log.md`
- Codex 与 Claude Code 具有**清晰区分的触发方式**
- Codex 支持通过 `$session-log + 动作描述` 触发
- Claude Code 支持通过 `/session-log + 动作参数` 触发
- 文件头部存在可持续更新的会话概览
- 开启记录模式后，每轮有效协作都能追加一条结构化摘要
- 历史记录不会被覆盖
- 用户打开文件后能快速看到：
  1. 当前整体进度
  2. 最近一次更新
  3. 各轮问题与解决过程
